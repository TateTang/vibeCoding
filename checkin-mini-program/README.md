# 打卡小程序数据结构与存储方案

## 1. 数据结构

### 1.1 打卡项目 `projects`

推荐数据结构：

```json
{
  "id": "proj_001",
  "name": "每天学习",
  "target": 30,
  "created_at": "2026-05-12T10:00:00+08:00"
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 是 | 项目唯一 ID，推荐使用 UUID 或带前缀字符串 |
| `name` | string | 是 | 项目名称，如“每天学习” |
| `target` | number | 是 | 目标天数，必须大于 0 |
| `created_at` | string | 是 | 创建时间，推荐 ISO 8601 格式 |

### 1.2 打卡记录 `checkin_records`

推荐数据结构：

```json
{
  "id": "rec_001",
  "project_id": "proj_001",
  "date": "2026-05-12",
  "note": "今天完成了 2 小时阅读",
  "created_at": "2026-05-12T21:30:00+08:00"
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 是 | 打卡记录唯一 ID |
| `project_id` | string | 是 | 关联的项目 ID |
| `date` | string | 是 | 打卡日期，格式 `YYYY-MM-DD` |
| `note` | string \| null | 否 | 打卡备注 |
| `created_at` | string | 是 | 记录创建时间，推荐 ISO 8601 格式 |

## 2. 关系与约束

- 一个项目可以有多条打卡记录，关系为 `1:N`。
- `project_id` 必须关联到一个已存在的项目。
- 同一个项目在同一天通常只允许一条打卡记录，建议加唯一约束：`(project_id, date)`。
- 删除项目时，建议级联删除对应打卡记录，避免脏数据。

## 3. 推荐存储方案

### 方案 A：本地存储，适合 MVP

适用场景：

- 先快速做出小程序原型
- 不需要登录
- 不要求多设备同步

推荐存储键：

- `checkin_projects`
- `checkin_records`

建议保存结构：

```json
{
  "projects": [],
  "records": []
}
```

优点：

- 开发快，接入成本最低
- 不依赖后端

缺点：

- 卸载小程序或清缓存可能丢数据
- 无法跨设备同步
- 后续多人或账号体系扩展困难

### 方案 B：云端数据库，适合正式版本

适用场景：

- 需要用户登录
- 需要多设备同步
- 后续可能扩展统计、排行榜、分享等功能

推荐做法：

- 小程序前端负责展示和提交流程
- 服务端或云函数负责写入数据库
- 数据库存 `projects` 和 `checkin_records` 两张表

推荐数据库：

- 微信云开发数据库：适合纯小程序方案，上手快
- MySQL / PostgreSQL：适合后续独立后端
- SQLite：适合本地应用或轻量服务

## 4. 我的建议

如果你现在的目标是尽快开始开发，建议这样走：

1. 第一阶段先用本地存储，把项目创建、打卡、列表展示跑通。
2. 数据结构从一开始就按关系型表设计，这样后续迁移到云端几乎不用改模型。
3. 从第一天开始保证 `project_id + date` 唯一，避免重复打卡问题。

## 5. 后续可扩展字段

如果后面要继续做完整产品，可以考虑补这些字段：

### `projects`

- `updated_at`：项目更新时间
- `status`：项目状态，如进行中、已完成
- `current_count`：当前已打卡天数，可缓存统计值

### `checkin_records`

- `updated_at`：记录更新时间
- `images`：打卡配图
- `mood`：当天状态标签
